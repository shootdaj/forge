<task type="auto">
  <name>Task 1: Git workflow utilities with unit tests</name>
  <files>src/cli/git.ts, src/cli/git.test.ts</files>
  <action>
Create `src/cli/git.ts` with the following git workflow functions. All functions shell out to git via `child_process.execSync` (or `spawnSync` for error handling). All functions accept an optional `cwd` parameter defaulting to `process.cwd()`, and an optional `execFn` parameter for testing (injectable executor function with same signature as execSync).

1. `getCurrentBranch(opts?)`: Returns current branch name via `git rev-parse --abbrev-ref HEAD`.

2. `isOnBranch(branchName, opts?)`: Returns boolean -- checks if current branch matches.

3. `createPhaseBranch(phaseNumber: number, opts?)`: Creates and checks out `phase-{phaseNumber}` from main. If already on that branch, no-op. If branch already exists, checks it out. Uses `git checkout -b phase-{phaseNumber} main` or `git checkout phase-{phaseNumber}`.

4. `commitWithReqId(requirementIds: string[], message: string, files: string[], phaseNumber: number, opts?)`: Stages the specified files via `git add`, then commits with format `feat({reqIds}): {message}` where reqIds is comma-separated. The commit body includes `Requirement: {reqIds}\nPhase: {phaseNumber}`. Uses heredoc-style commit via `-m` flag. If no files provided, uses `git add -A`. Returns the commit hash via `git rev-parse HEAD`.

5. `mergePhaseBranch(phaseNumber: number, opts?)`: Switches to main (`git checkout main`), merges `phase-{phaseNumber}` with `--no-ff` flag and message `merge: phase {phaseNumber} verified`, then deletes the phase branch (`git branch -d phase-{phaseNumber}`).

6. `hasUncommittedChanges(opts?)`: Returns boolean via `git status --porcelain`.

7. `branchExists(branchName, opts?)`: Returns boolean via `git branch --list {branchName}`.

All functions throw descriptive errors on git command failure (catch execSync errors, extract stderr).

Create `src/cli/git.test.ts` with unit tests that use a temporary git repo (created in `beforeEach` with `git init`, `git commit --allow-empty -m "init"`). Test:
- createPhaseBranch creates the branch and switches to it
- createPhaseBranch is idempotent (no error if already on branch)
- commitWithReqId formats commit message correctly with requirement IDs
- commitWithReqId includes phase number in body
- mergePhaseBranch merges to main and deletes branch
- getCurrentBranch returns correct branch name
- hasUncommittedChanges detects modified files
- branchExists returns true/false correctly
- Error handling: commitWithReqId throws if nothing to commit (no staged changes, no files)

Use `node:child_process` execSync with `{ cwd: tmpDir, encoding: 'utf-8' }` in tests to set up real temp git repos. Use `node:os` tmpdir + `node:fs` mkdtemp for temp directories. Clean up in afterEach.

<task type="auto">
  <name>Task 2: Testing infrastructure (traceability + methodology injection) with unit tests</name>
  <files>src/cli/traceability.ts, src/cli/traceability.test.ts</files>
  <action>
Create `src/cli/traceability.ts` with TEST_GUIDE.md management and test pyramid enforcement functions. All filesystem operations use an injectable `fs` parameter (defaulting to `node:fs`) for testability.

1. `createTestGuide(requirements: Array<{id: string, description: string}>, outputPath: string, fs?)`: Creates the initial TEST_GUIDE.md with the traceability matrix header and one row per requirement. Columns: Req ID, Requirement, Unit Tests, Integration Tests, Scenario Tests. Initial test columns are empty (TBD).

2. `updateTestGuide(phaseRequirementIds: string[], testMappings: Array<{reqId: string, tier: 'unit'|'integration'|'scenario', testName: string}>, guidePath: string, fs?)`: Reads existing TEST_GUIDE.md, finds rows matching the reqIds, appends test names to the appropriate tier column. If a test name already exists in that cell, skip (idempotent). Writes back the updated file.

3. `parseTestGuide(guidePath: string, fs?)`: Reads TEST_GUIDE.md and returns a structured array of `{reqId: string, description: string, unitTests: string[], integrationTests: string[], scenarioTests: string[]}`. Parses the markdown table by splitting on `|`.

4. `verifyTestCoverage(guidePath: string, fs?)`: Reads and parses TEST_GUIDE.md, returns `{covered: string[], uncovered: string[], missingTiers: Array<{reqId: string, missing: ('unit'|'integration'|'scenario')[]}>}`. A requirement is "covered" if it has at least one test in each tier. Returns the gap analysis.

5. `enforceTestPyramid(currentCounts: {unit: number, integration: number, scenario: number}, previousCounts: {unit: number, integration: number, scenario: number})`: Returns `{passed: boolean, violations: string[]}`. Checks: (a) unit >= integration >= scenario (pyramid shape), (b) each count increased from previous (new code must have tests). Violations describe what failed.

6. `injectTestingMethodology(claudeMdPath: string, config: {testNaming: string, tiers: string[], requirementPrefix: string}, fs?)`: Reads the target project's CLAUDE.md, checks if testing methodology section already exists (idempotent via marker `<!-- FORGE:TESTING_METHODOLOGY -->`). If not present, appends a section with test naming conventions (e.g., `Test<Component>_<Behavior>`), tier definitions (unit/integration/scenario), requirement reference format, and test pyramid expectations. Writes back.

7. `generateTestingMethodologyBlock(config)`: Pure function returning the markdown string for the testing methodology section, wrapped in the FORGE marker comments.

Create `src/cli/traceability.test.ts` with unit tests using in-memory filesystem (Map-based, matching the pattern from Phase 5/6 tests). Test:
- createTestGuide creates correct markdown table with all requirements
- updateTestGuide appends test names to correct columns
- updateTestGuide is idempotent (no duplicate test names)
- parseTestGuide correctly parses the markdown table
- verifyTestCoverage identifies uncovered requirements and missing tiers
- enforceTestPyramid passes when pyramid shape holds and counts increase
- enforceTestPyramid fails when pyramid is inverted or counts don't increase
- injectTestingMethodology appends to CLAUDE.md
- injectTestingMethodology is idempotent (no duplicate injection)
- generateTestingMethodologyBlock produces correct markdown with markers

Use the in-memory fs pattern: `const files = new Map<string, string>()` with `existsSync`, `readFileSync`, `writeFileSync`, `mkdirSync` implementations backed by the Map.
