   - PHA-07: Instructs the agent to execute the plan

   - GAP-02: Creates a targeted fix prompt based on diagnosis
   - Specifies exact files to change, what fix to apply, what to retest
   - NOT a full re-execution -- just the targeted fix

7. `buildTestGapPrompt(planContent: string, missingCoverageComponents: string[]): string`
   - PHA-10: Instructs agent to add tests for components missing coverage
   - Lists specific components that need tests

8. `buildReportPrompt(phaseNumber: number, verificationContent: string, gapsContent: string | null, contextContent: string): string`
   - Instructs agent to generate PHASE_REPORT.md summarizing the phase
   - Includes phase goals, test results, issues encountered, budget used

All prompt builders are pure functions (string in, string out). They read checkpoint files passed as string parameters -- they do NOT read files themselves.

**Then, create all substep files in `src/phase-runner/substeps/`:**

Each substep follows the same pattern:
- Takes `phaseNumber`, `phaseDir`, and `ctx: PhaseRunnerContext`
- Reads relevant checkpoint files from prior substeps (via fs)
- Calls `runStep()` with the appropriate prompt
- Writes its checkpoint file on success
- Returns void (or throws on unrecoverable error)

- Calls runStep() with buildPlanPrompt
- Verify callback: check PLAN.md exists in phaseDir
- On success: PLAN.md is the checkpoint

**`execute.ts`** -- `executePlan(phaseNumber, phaseDir, ctx, mockInstructions?)` (PHA-07):
- Reads PLAN.md and CONTEXT.md from phaseDir
- Calls runStepWithCascade() with buildExecutePrompt (this is the long step, needs cascade)
- onFailure callback: builds retry prompt with error context
- Verify callback: basic file existence check (expected files from plan)
- On success: writes EXECUTION_MARKER to phaseDir

**`verify-build.ts`** -- `verifyBuild(phaseNumber, phaseDir, ctx): Promise<VerificationReport>` (PHA-08):
- Calls runVerifiers() from Phase 4 with the project config
- Writes VERIFICATION.md with the verification report content (formatted as markdown)
- Returns the VerificationReport for gap closure decision

**`gap-closure.ts`** -- `runGapClosure(phaseNumber, phaseDir, initialReport, ctx)` (PHA-09, GAP-01, GAP-02, GAP-03):
- **MAX_GAP_CLOSURE_ROUNDS = 2** (from CONTEXT.md, locked)
- Loop for up to 2 rounds:
  - `diagnoseFailures(report, phaseDir, ctx)`: calls runStep() with buildDiagnosisPrompt and `outputSchema` for GapDiagnosis structured output. Returns GapDiagnosis.
  - `executeTargetedFix(diagnosis, phaseNumber, ctx)`: calls runStepWithCascade() with buildFixPrompt. Uses limited retries (maxRetries: 1) since gap closure is itself a retry mechanism.
  - Re-verify: calls runVerifiers() again
  - If passed: write GAPS.md with resolution history, return
- If still failing after 2 rounds: write GAPS.md with remaining failures, return
- The `diagnoseFailures` function produces structured output with the GapDiagnosis schema (using outputSchema on StepOptions)

**`test-gaps.ts`** -- `fillTestGaps(phaseNumber, phaseDir, ctx)` (PHA-10):
- Reads PLAN.md and checks which components lack tests (using detectMissingTestTasks from plan-verification)
- If missing tests found: calls runStep() with buildTestGapPrompt
- Verify callback: re-run coverage verifier to check new tests exist

**`docs.ts`** -- `generatePhaseReport(phaseNumber, phaseDir, ctx)`:
- Reads VERIFICATION.md, GAPS.md (if exists), CONTEXT.md
- Calls runStep() with buildReportPrompt
- Verify callback: check PHASE_REPORT.md exists
- On success: PHASE_REPORT.md is the checkpoint

All substep functions accept the `fs` abstraction from `PhaseRunnerContext.fs` (defaulting to real `node:fs`). File reads within substeps use `ctx.fs?.readFileSync ?? fs.readFileSync`.

Do NOT call the SDK directly in any substep -- all SDK interaction goes through runStep() and runStepWithCascade().

**Create `src/phase-runner/phase-runner.ts`** -- the main orchestrator (PHA-01):

`runPhase(phaseNumber: number, ctx: PhaseRunnerContext, options?: { mockInstructions?: string; requirementIds?: string[] }): Promise<PhaseResult>`

This is the checkpoint sequencer. Flow:

1. **Resolve phase directory** via resolvePhaseDir(phaseNumber)
2. **Detect existing checkpoints** via detectCheckpoints(phaseDir)
3. **Update state**: set phase status to 'in_progress', set startedAt
4. **Get phase requirement IDs**: from options.requirementIds (or empty array if not provided)
5. **Context gathering** (skip if contextDone): call gatherContext()
6. **Plan creation** (skip if planDone): call createPlan()
7. **Plan verification** (always run -- idempotent):
   - Read PLAN.md, call verifyPlanCoverage with requirement IDs
   - If hasTestTasks is false: call injectTestTasks, write updated PLAN.md
   - If missingRequirements.length > 0: call runStep with buildReplanPrompt (PHA-06), then re-verify (max 1 re-plan attempt)
   - If still failing after re-plan: return PhaseResult { status: 'failed', reason: 'Plan verification failed' }
8. **Execution** (skip if executionDone): call executePlan(phaseNumber, phaseDir, ctx, options.mockInstructions)
9. **Verification** (skip if verificationDone):
   - Call verifyBuild() -> returns VerificationReport
   - If !report.passed: call fillTestGaps() if coverage issues, then runGapClosure()
10. **Gap closure** (skip if gapsDone): Already handled in step 9
11. **Documentation** (skip if reportDone): call generatePhaseReport()
12. **Update state**: set phase status to 'completed', set completedAt, update test results
13. **Return** PhaseResult { status: 'completed', report: PHASE_REPORT.md path }

Error handling:
- Wrap each substep in try/catch
- On error in any substep: update state to 'partial', return PhaseResult { status: 'partial', completedSubsteps: [...], lastError: error.message }
- Budget exceeded errors (from runStep): update state to 'partial', return immediately
