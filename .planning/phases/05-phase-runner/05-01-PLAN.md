Create `src/phase-runner/types.ts` with all type definitions for the phase runner module:

1. **PhaseSubstep enum**: `'context' | 'plan' | 'verify-plan' | 'execute' | 'verify-build' | 'gap-closure' | 'docs'`

2. **CheckpointState interface**: Boolean flags for each substep completion:
   - `contextDone`, `planDone`, `executionDone`, `verificationDone`, `gapsDone`, `reportDone`

3. **PlanVerificationResult interface** (from RESEARCH.md):
   - `passed: boolean`
   - `coveredRequirements: string[]`
   - `missingRequirements: string[]`
   - `hasTestTasks: boolean`
   - `missingTestTasks: string[]`
   - `executionOrderValid: boolean`
   - `hasSuccessCriteria: boolean`
   - `scopeCreep: string[]`

4. **GapDiagnosis interface** (structured output from root cause diagnosis):
   - `category: 'wrong_approach' | 'missing_dependency' | 'integration_mismatch' | 'requirement_ambiguity' | 'environment_issue'`

   - `affectedFiles: string[]`

   - `diagnosis: GapDiagnosis`

   - `{ status: 'failed'; reason: string; gapsRemaining?: string[] }`
   - `{ status: 'partial'; completedSubsteps: PhaseSubstep[]; lastError: string }`

7. **PhaseRunnerContext interface** (dependency injection container):
   - `config: ForgeConfig`
   - `stateManager: StateManager`
   - `stepRunnerContext: StepRunnerContext`
   - `costController: CostController`
   - `fs?: { existsSync, readFileSync, writeFileSync, mkdirSync }` (for testing)

  <name>Task 2: Implement plan verification and test task injection (pure functions)</name>
  <files>src/phase-runner/plan-verification.ts, src/phase-runner/plan-verification.test.ts</files>
  <action>
Create `src/phase-runner/plan-verification.ts` with pure function plan verification:

1. **`parsePlanRequirements(planContent: string): string[]`** (PHA-04)
   - Extract requirement IDs from plan content using regex
   - Pattern: `/\b([A-Z]+-\d+)\b/gi` -- matches PHA-01, GAP-03, STEP-01, etc.
   - Case-insensitive matching, normalize to uppercase
   - Deduplicate using Set
   - Return sorted array of unique requirement IDs

2. **`verifyPlanCoverage(planContent: string, requiredIds: string[]): PlanVerificationResult`** (PHA-04)
   - Call parsePlanRequirements to get covered IDs
   - Compute missingRequirements: required IDs not found in plan
   - Check for test tasks: look for keywords like "test", "tests", "testing", "unit test", "integration test", "scenario test" in task sections
   - Check for success criteria: look for "Success" or "Verification" or "verify" or "done" sections
   - Detect scope creep: requirement IDs in plan that are NOT in the required list (filter out common prefixes like TEST-, GEN-)
   - Check execution order: verify tasks are numbered sequentially (basic heuristic)
   - `passed` is true when: missingRequirements.length === 0 AND hasTestTasks
   - Return full PlanVerificationResult

3. **`injectTestTasks(planContent: string, components: string[]): string`** (PHA-05)
   - Appends a test tasks section at the end of the plan content (before </tasks> closing tag if present, otherwise at the end)
   - Each component gets a test task block with unit, integration, and scenario test requirements
   - Adds `<!-- FORGE:INJECTED_TEST_TASKS -->` marker for traceability
   - Task numbering: detect highest existing task number and continue from there
   - Returns the modified plan content as a string

4. **`detectMissingTestTasks(planContent: string): string[]`** (PHA-05)
   - Parse plan to find component/module names mentioned in task `<files>` sections
   - Check if there are corresponding test tasks for those components
   - Return list of component names that lack test tasks

Then create `src/phase-runner/plan-verification.test.ts` with unit tests:

- `TestPlanVerification_ParseRequirements_Standard`: Extracts PHA-01, PHA-02 from normal plan
- `TestPlanVerification_ParseRequirements_CaseInsensitive`: Extracts `pha-01` as `PHA-01`
- `TestPlanVerification_ParseRequirements_MultipleFormats`: Extracts from `(PHA-01)`, `[PHA-01]`, `PHA-01:`, etc.
- `TestPlanVerification_ParseRequirements_Deduplicates`: Same ID appearing twice returns once
- `TestPlanVerification_VerifyCoverage_AllCovered`: All required IDs present -> passed: true
- `TestPlanVerification_VerifyCoverage_MissingRequirements`: Missing IDs -> passed: false, lists missing
- `TestPlanVerification_VerifyCoverage_MissingTestTasks`: No test keywords -> passed: false, hasTestTasks: false
- `TestPlanVerification_VerifyCoverage_ScopeCreep`: Extra IDs not in required list detected
- `TestPlanVerification_VerifyCoverage_HasSuccessCriteria`: Detects success criteria presence
- `TestPlanVerification_InjectTestTasks_AppendsSection`: Injected content appears in output
- `TestPlanVerification_InjectTestTasks_MarkerPresent`: FORGE:INJECTED_TEST_TASKS marker in output
- `TestPlanVerification_InjectTestTasks_PreservesExisting`: Original content unchanged before injection point
- `TestPlanVerification_InjectTestTasks_ContinuesNumbering`: Task numbers continue from existing tasks
- `TestPlanVerification_DetectMissingTests_FindsMissing`: Components without test tasks detected

Use realistic plan content snippets (mimicking GSD PLAN.md format) as test fixtures. These are pure function tests -- no fs or mocking needed.
