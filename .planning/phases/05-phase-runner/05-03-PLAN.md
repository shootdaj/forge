Create `test/scenarios/phase-runner.test.ts` with end-to-end scenario tests that exercise the full phase runner from the user's perspective.

These tests treat the phase runner as a black box: call runPhase() with a context, verify the PhaseResult and side effects. Mock only the SDK's executeQueryFn (since we can't make real API calls in tests).

**Setup**: Reuse or extend the integration test helper. The key difference: scenario tests verify observable outcomes (PhaseResult, files on disk, state), not internal wiring.

**Scenarios:**

1. `TestPhaseRunner_Scenario_HappyPath` (PHA-01 -- the golden path):
   - Run runPhase(1, ctx) with mocked SDK that succeeds at every step
   - Assert PhaseResult.status === 'completed'
   - Assert all checkpoint files exist in phase directory
   - Assert state shows phase completed
   - This tests the full cycle: context -> plan -> verify-plan -> execute -> verify-build -> docs

2. `TestPhaseRunner_Scenario_ResumeFromCheckpoint` (PHA-12):
   - Pre-create CONTEXT.md and PLAN.md (with valid content including requirement IDs) in the phase directory
   - Run runPhase() -- should skip context and plan creation
   - Assert executeQueryFn NOT called for context or plan prompts
   - Assert execution and verification still run
   - Assert PhaseResult.status === 'completed'

3. `TestPhaseRunner_Scenario_ResumeFromExecution` (PHA-12):
   - Pre-create CONTEXT.md, PLAN.md, and EXECUTION_MARKER
   - Run runPhase() -- should skip context, plan, and execution
   - Assert only verification + docs substeps run
   - Assert PhaseResult.status === 'completed'

4. `TestPhaseRunner_Scenario_GapClosureSuccess` (PHA-09, GAP-01, GAP-02, GAP-03):
   - Mock runVerifiers to fail on first call, pass on second (after fix)
   - Mock diagnosis to return { category: 'wrong_approach', suggestedFix: '...' }
   - Run runPhase()
   - Assert diagnosis was called
   - Assert targeted fix was executed (not full phase re-run)
   - Assert GAPS.md exists with 'resolved' status
   - Assert PhaseResult.status === 'completed'

5. `TestPhaseRunner_Scenario_GapClosureExhausted` (PHA-09, GAP-01):
   - Mock runVerifiers to always fail
   - Mock diagnosis to return valid diagnosis each time
   - Run runPhase()
   - Assert gap closure ran exactly 2 rounds (MAX_GAP_CLOSURE_ROUNDS)
   - Assert GAPS.md exists with 'unresolved' status
   - Assert PhaseResult reflects completion (gap closure doesn't block phase report)

6. `TestPhaseRunner_Scenario_PlanVerificationWithInjection` (PHA-04, PHA-05):
   - Mock SDK to write a PLAN.md with all requirements but no test tasks
   - Run runPhase()
   - Assert injectTestTasks was called (PLAN.md now contains FORGE:INJECTED_TEST_TASKS marker)
   - Assert execution proceeds after injection

7. `TestPhaseRunner_Scenario_PlanVerificationReplanning` (PHA-06):
   - Mock SDK to write a PLAN.md missing some requirement IDs
   - Mock re-plan step to produce a PLAN.md with all requirements
   - Run runPhase()
   - Assert re-plan step was called with feedback about missing requirements
   - Assert execution proceeds with the corrected plan

8. `TestPhaseRunner_Scenario_BudgetExceeded` (PHA-01 error path):
   - Set config.maxBudgetTotal very low (e.g., 0.01)
   - Set state.totalBudgetUsed to 0.01 (already at limit)
   - Run runPhase()
   - Assert PhaseResult.status === 'partial' or 'failed'
   - Assert state NOT set to 'completed'

9. `TestPhaseRunner_Scenario_MultipleCheckpointsCreated` (PHA-11):
   - Run full phase to completion
   - Assert existence of: CONTEXT.md, PLAN.md, .execution-complete, VERIFICATION.md, PHASE_REPORT.md
   - Assert each file has non-empty content (not zero-length files)
