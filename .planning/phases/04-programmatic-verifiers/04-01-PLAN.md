<objective>
Implement the type system, shared utilities, config schema updates, and all 8 individual verifier functions with comprehensive unit tests.

Purpose: Build every verifier as a standalone, independently testable function following the locked Verifier interface. Each verifier delegates hard work to established CLI tools (tsc, vitest, eslint, docker compose, git) and only parses output. This plan creates all the pieces; the next plan wires them into the registry.

Output: 8 working verifiers, each with unit tests proving pass/fail/skip behavior.
</objective>

Key interfaces from CONTEXT.md (locked, implement exactly):
```typescript
interface VerifierResult {
  passed: boolean;
  verifier: string;       // e.g., "files", "tests", "typecheck"
  details: string[];      // Human-readable descriptions
  errors: string[];       // Specific failure messages with paths/line numbers
}

interface VerifierConfig {
  cwd: string;
  forgeConfig: ForgeConfig;
  expectedFiles?: string[];
  gitRef?: string;
}

type Verifier = (config: VerifierConfig) => Promise<VerifierResult>;
```

    - `details`: "TypeScript compilation: X error(s)"

    **1. Coverage verifier (`src/verifiers/coverage.ts`) — VER-05:**
    - Export `coverageVerifier: Verifier`
    - Get new source files: run `git diff --name-only --diff-filter=A ${config.gitRef ?? "main"}...HEAD` via `execWithTimeout`
    - Filter to source files: match `.ts`, `.tsx`, `.js`, `.jsx` in `src/` directory
    - Exclude: `*.test.ts`, `*.spec.ts`, `*.d.ts`, `index.ts` (barrel files), `types.ts` (pure type files)
    - For each source file, check for corresponding test file in order:
      1. Co-located: `src/foo.test.ts` (replace `.ts` with `.test.ts`)
      2. Co-located alt: `src/foo.spec.ts`
      3. Separate dir: `test/{relative}/foo.test.ts`
      4. Separate dir tiered: `test/unit/{relative}/foo.test.ts`
    - Check existence via `fs.existsSync(path.resolve(config.cwd, testPath))`
    - `passed`: all new source files have at least one test file
    - `details`: list each source file and its test file (or "MISSING")
    - `errors`: list source files without test files
    - If git command fails (not a git repo, no commits), return `skippedResult("coverage", "Git not available or no commits")`

    **2. Observability verifier (`src/verifiers/observability.ts`) — VER-06:**
    - Export `observabilityVerifier: Verifier`
    - Three-check heuristic:
      a. **Health endpoint**: grep all `.ts`/`.js` files in `src/` for patterns:
         `/app\.(get|use)\s*\(\s*['"\`]\/health/`, `/router\.(get|use)\s*\(\s*['"\`]\/health/`, `/\.get\s*\(\s*['"\`]\/healthz?['"\`]/`, `/healthCheck/`
         Use `execWithTimeout("grep -rl ...")` or read files and match
      b. **Structured logging**: check for imports of pino, winston, bunyan, or `createLogger`/`Logger(` patterns
      c. **Error logging**: check for `.error(` or `log.error(` in catch-like contexts
    - Only FAIL if health endpoint is missing (most critical and reliably detectable). Warn on missing structured logging or error logging.
    - `passed`: health endpoint found (or skip if no src/ directory)
    - `details`: report each check result individually
    - `errors`: only if health endpoint missing

    **3. Docker verifier (`src/verifiers/docker.ts`) — VER-07:**
    - Export `dockerVerifier: Verifier`
    - Get compose file from `config.forgeConfig.testing.dockerComposeFile` (default: `docker-compose.test.yml`)
    - Check prerequisite: `fs.existsSync(path.resolve(config.cwd, composeFile))`. If missing, return `skippedResult("docker", "No ${composeFile} found")`
    - Run in try/finally:
      - `docker compose -f ${composeFile} up --wait` with 120s timeout
      - Capture result
      - In finally: `docker compose -f ${composeFile} down --volumes --remove-orphans` with 30s timeout (Pitfall 6)
    - `passed`: up --wait exitCode === 0
    - `details`: "Docker compose smoke test: {passed|failed}"
    - `errors`: if failed, include stderr from up command

    **4. Deployment verifier (`src/verifiers/deployment.ts`) — VER-08:**
    - Export `deploymentVerifier: Verifier`
    - Check prerequisites:
      a. `Dockerfile` exists in cwd. If not, skip.
    - Checks:
      a. **Dockerfile builds**: run `docker build --check .` or `docker build -f Dockerfile --no-cache --target=builder .` (dry run). Actually, simplest: just check Dockerfile exists and is non-empty. Full build check is too expensive for a verifier. Report existence.
      b. **Env var consistency**: If `.env.example` exists, parse variable names (lines not starting with `#`, split on `=`, take left side). Parse Dockerfile for `ENV` and `ARG` declarations. Report variables in `.env.example` but not in Dockerfile (informational, not a hard fail — many env vars are runtime-injected).
    - `passed`: Dockerfile exists (minimum bar). Env var warnings are details, not errors.
    - `details`: Dockerfile status, env var comparison results
    - `errors`: only if Dockerfile is missing (but that's already a skip case)

    **5. Unit tests for each verifier:**

    - TestCoverageVerifier_ExcludesIndexAndTypes: index.ts and types.ts excluded
    - TestCoverageVerifier_GitNotAvailable: git command fails → skipped
    - TestCoverageVerifier_ColocatedAndSeparatePatterns: verifies both src/foo.test.ts and test/foo.test.ts are checked
