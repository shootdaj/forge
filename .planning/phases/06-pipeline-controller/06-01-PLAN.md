Create `src/pipeline/dependency-graph.ts` with pure functions:

1. `parseRoadmapPhases(roadmapContent: string): PipelinePhase[]` -- parses ROADMAP.md content to extract phase entries. Regex-based: looks for `### Phase N:` headers, extracts "Depends on" field, "Requirements" field, and phase name. Returns array of PipelinePhase objects.

2. `buildDependencyGraph(phases: PipelinePhase[]): Map<number, Set<number>>` -- builds adjacency list: key = phase number, value = set of phases it depends on. Validates no self-dependencies. Throws on circular dependency (detect via DFS with visited/recursion-stack).

3. `topologicalSort(graph: Map<number, Set<number>>): number[][]` -- Kahn's algorithm: returns array of arrays where each inner array is a "wave" of independent phases. Phases with no unresolved deps go in wave 1. Phases whose deps are all in earlier waves go in wave 2, etc. This is the execution order.

4. `getExecutionWaves(roadmapContent: string): { waves: number[][]; phases: PipelinePhase[] }` -- convenience: chains parse -> build graph -> toposort.

Create `src/pipeline/mock-manager.ts` -- a class that manages the mock registry:

```typescript
class MockManager {
  constructor(private stateManager: StateManager) {}

  // Register a mock entry for a service (MOCK-01, MOCK-02)
  async registerMock(serviceName: string, entry: MockEntry): Promise<void>
  // Updates state.mockRegistry[serviceName] = entry

  // Get all registered mocks (MOCK-02)
  async getMockRegistry(): Promise<Record<string, MockEntry>>
  // Returns state.mockRegistry

  // Get a single mock entry
  async getMock(serviceName: string): Promise<MockEntry | undefined>

  // Build mock instructions for a phase execution prompt (PIPE-02)
  // Returns a string that tells the agent to use the 4-file pattern
  buildMockInstructions(services: ServiceDetection[]): string
  // Generates text like:
  // "For each external service, create 4 files:
  //  1. src/services/{name}.ts — TypeScript interface
  //  2. src/services/{name}.mock.ts — Mock implementation (// FORGE:MOCK tag)
  //  3. src/services/{name}.real.ts — Real implementation (stub, filled in Wave 2)
  //  4. src/services/{name}.factory.ts — Factory returning mock or real based on env
  //  Mock and real MUST implement the same interface (MOCK-04)."

  // Detect external services from phase description (PIPE-02)
  // Uses keyword matching: "stripe", "aws", "sendgrid", "twilio", "openai",
  // "oauth", "smtp", "database", "redis", "s3", "firebase", etc.
  detectExternalServices(phaseDescription: string, phaseNumber: number): ServiceDetection[]

  // Build swap prompt for Wave 2 (MOCK-03)
  // Given the full mock registry, produces a prompt telling the agent
  // to replace each mock with a real implementation using credentials
  buildSwapPrompt(registry: Record<string, MockEntry>, credentials: Record<string, string>): string

  // Validate that a mock entry has all 4 files (for verification)
  validateMockEntry(entry: MockEntry, fs?: { existsSync: (p: string) => boolean }): { valid: boolean; missing: string[] }
}
```

Key implementation details:
- `registerMock` and `getMockRegistry` use `stateManager.update()` and `stateManager.load()` respectively
- `detectExternalServices` is a pure function that scans text for known service keywords. Returns ServiceDetection with appropriate credentialsNeeded for each service (e.g., Stripe needs STRIPE_SECRET_KEY, STRIPE_WEBHOOK_SECRET)
- `buildMockInstructions` produces a string appended to phase execution prompts during Wave 1
- `buildSwapPrompt` produces a detailed prompt listing each mock file and what credentials to use
- All mock files MUST have `// FORGE:MOCK -- swap for real in Wave 2` comment (MOCK-01)
- `validateMockEntry` checks that all 4 file paths in the entry are non-empty strings; optionally checks files exist on disk
